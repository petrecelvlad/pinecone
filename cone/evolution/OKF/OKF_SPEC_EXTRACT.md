# Open Knowledge Format — Core Technical Extract

Source: [GoogleCloudPlatform/knowledge-catalog/okf/SPEC.md](https://github.com/GoogleCloudPlatform/knowledge-catalog/blob/main/okf/SPEC.md)
Version: 0.1 — Draft
Published: 2026-06-12

This document extracts OKF's pure concepts for evaluation. Nothing here is adapted to pinecone — this is the spec as-is.

---

## What OKF Is

A format for representing knowledge — metadata, context, and curated insight that surrounds data and systems. Designed to be authored by people, generated by agents, exchanged across organizations, and consumed by both.

The format is a directory of markdown files with YAML frontmatter. No schema registry, no central authority, no required tooling.

## Design Values

- Readable by humans without tooling
- Parseable by agents without bespoke SDKs
- Diffable in version control
- Portable across tools, organizations, and time
- Minimally opinionated — standardizes only what's needed for self-description

## Non-Goals

- No fixed taxonomy of concept types
- No prescribed storage, serving, or query infrastructure
- Does not replace domain-specific schemas (Avro, Protobuf, OpenAPI) — OKF *references* them

---

## Core Terminology

| Term | Definition |
|---|---|
| **Knowledge Bundle** | A self-contained, hierarchical collection of knowledge documents. The unit of distribution. |
| **Concept** | A single unit of knowledge. One markdown document. May describe a tangible asset (table, API) or an abstract idea (metric, process). |
| **Concept ID** | The file path within the bundle, `.md` suffix removed. `tables/users.md` → `tables/users` |
| **Frontmatter** | YAML metadata block delimited by `---` at file top. |
| **Body** | Everything after the frontmatter. Standard markdown. |
| **Link** | A standard markdown link from one concept to another. Expresses relationships. |
| **Citation** | A link from a concept to an external source supporting a claim in the body. |

## Bundle Structure

A bundle is a directory tree of markdown files. Directory structure is domain-independent — producers organize however makes sense.

```
path/to/bundle/
├── index.md              # Optional. Directory listing for progressive disclosure.
├── log.md                # Optional. Chronological history of updates.
├── <concept>.md          # A concept at the bundle root.
└── <subdirectory>/       # Subdirectories organize concepts into groups.
    ├── index.md
    ├── <concept>.md
    └── <subdirectory>/
```

Distribution: git repo (recommended), tarball/zip, or subdirectory within a larger repo.

### Reserved Filenames

| Filename | Purpose |
|---|---|
| `index.md` | Directory listing for progressive disclosure (§6) |
| `log.md` | Update history (§7) |

All other `.md` files are concept documents.

## Concept Documents

Every concept is a UTF-8 markdown file with two parts: YAML frontmatter + markdown body.

### Frontmatter Schema

```yaml
---
type: <Type name>                  # REQUIRED — only required field
title: <Display name>              # Recommended
description: <One-line summary>    # Recommended
resource: <Canonical URI>          # Recommended — for asset-bound concepts
tags: [tag, tag, …]                # Optional
timestamp: <ISO 8601 datetime>     # Optional — last meaningful change
# … any additional producer-defined key/value pairs
---
```

**Required:** `type` only. A short string identifying the kind of concept. Used for routing, filtering, presentation. Values are NOT registered centrally. Consumers MUST tolerate unknown types (treat as generic).

**Recommended (priority order):** `title` → `description` → `resource` → `tags` → `timestamp`

**Extensions:** Producers MAY include any additional keys. Consumers SHOULD preserve unknown keys when round-tripping and MUST NOT reject documents with unrecognized fields.

### Body Conventions

Standard markdown. Structural markdown (headings, lists, tables, code blocks) preferred over freeform prose — aids both human reading and agent retrieval.

No required sections. Conventional section headings:

| Heading | Purpose |
|---|---|
| `# Schema` | Structured description of an asset's columns/fields |
| `# Examples` | Concrete usage examples, often fenced code blocks |
| `# Citations` | External sources backing claims in the body |

## Cross-Linking (The Relationship Model)

Concepts link to other concepts using standard markdown links. This is how the graph is built.

### Link Forms

**Absolute (bundle-relative)** — recommended, stable when files move within subdirectory:
```markdown
See the [customers table](/tables/customers.md) for the join key.
```

**Relative:**
```markdown
See the [neighboring concept](./other.md).
```

### Link Semantics

A link from A to B asserts a *relationship*. The kind of relationship (parent/child, references, joins-with, depends-on) is conveyed by surrounding prose, NOT by the link syntax itself.

Consumers building a graph view treat all links as directed edges of an untyped relationship.

Consumers MUST tolerate broken links — a target that doesn't exist is not malformed, it may represent not-yet-written knowledge.

## Index Files (Progressive Disclosure)

`index.md` may appear in any directory. Enumerates contents so humans/agents see what's available before opening individual documents.

- No frontmatter (except optionally at bundle root for `okf_version`)
- Body uses sections grouping concepts under headings
- Entries SHOULD include the description from the linked concept's frontmatter
- May be generated automatically; consumers may synthesize one when absent

```markdown
# Section Heading

* [Title 1](relative-url-1) - short description
* [Title 2](relative-url-2) - short description

# Another Section

* [Subdirectory](subdir/) - short description
```

---

## Log Files (Change History)

Optional `log.md` at any hierarchy level. Flat list of date-grouped entries, newest first.

- Date headings MUST use ISO 8601 `YYYY-MM-DD`
- Leading bold word (`**Update**`, `**Creation**`, `**Deprecation**`) is convention, not required

---

## Citations

Sources listed under `# Citations` heading at document bottom, numbered:

```markdown
# Citations

[1] [BigQuery public dataset announcement](https://cloud.google.com/blog/...)
[2] [Internal data quality runbook](https://wiki.acme.internal/data/quality)
```

May be absolute URLs, bundle-relative paths, or paths into a `references/` subdirectory.

---

## Conformance (The Full Rule Set)

A bundle is conformant with OKF v0.1 if:

1. Every non-reserved `.md` file contains a parseable YAML frontmatter block
2. Every frontmatter block contains a non-empty `type` field
3. Every reserved filename (`index.md`, `log.md`) follows its defined structure when present

**That's it.** Everything else is soft guidance. Consumers MUST NOT reject a bundle because of:

- Missing optional frontmatter fields
- Unknown `type` values
- Unknown additional frontmatter keys
- Broken cross-links
- Missing `index.md` files

This permissive model is intentional — OKF stays useful as bundles grow, get refactored, and are partially generated by agents.

## Versioning

Version format: `<major>.<minor>`
- Minor bump: backward-compatible additions (new optional fields, new conventional headings)
- Major bump: may make breaking changes (renaming required fields, changing reserved filenames)

Bundles may declare version via `okf_version: "0.1"` in bundle-root `index.md` frontmatter (only place frontmatter is permitted in an `index.md`).

## Visualizer (Reference Tooling)

Google ships a `visualize` subcommand that renders any OKF bundle into a single self-contained HTML file:

- Uses Cytoscape.js for force-directed graph layout
- Uses marked for in-browser markdown rendering
- Bundle embedded as JSON blob
- Nodes colored by `type`, directed edges drawn from cross-links
- Click a node → see frontmatter + rendered body in detail panel
- Internal links rewired to navigate within the viewer
- Zero backend, zero install

## Relationship to Existing Patterns

OKF is close to:
- LLM "wiki" repositories (markdown + frontmatter as agent-readable knowledge)
- Personal knowledge tools (Obsidian, Notion) with hierarchical markdown + cross-links
- "Metadata as code" (catalog metadata alongside source code, not in a separate registry)

OKF differs primarily in being *specified* — pinning down the small rule set needed for interoperability without dictating tooling.
